## 1. 这份仓库在做什么

这是一套 **Agent Infra 的内核实现**，当前以 Go 语言实现了三个核心内核：

1. **Decision Kernel**（裁决内核）
   - 负责是否允许执行、是否要审批、是否 fail closed。
   - 负责策略规则、特征合同（Feature Signal Contract）、调度准入、发布准入等判定。

2. **Run Kernel**（运行内核）
   - 负责 run 生命周期状态机（RUNNING/PARKED/AWAITING_APPROVAL/...）。
   - 负责 step 推进、幂等、continuation、完整性链、TTL/Zombie 清理。

3. **Evidence Kernel**（证据内核）
   - 负责事件摄取、标准化、决策图、账本、回放包、根因包、完整性校验、归档、DSAR 删除。

`internal/httpapi` 是 transport 层，把 HTTP API 映射到三个内核，并负责把 Decision/Run 事件同步进 Evidence。

## 3. 三个 Kernel 如何协同

### 3.1 依赖方向（非常重要）

1. `Run -> Decision`
   - Run 推进前必须向 Decision 查询：
     - 决策是否已确认（owner-aware）
     - 决策引用是否绑定当前 run/step
     - ticket/receipt 是否有效

2. `Decision -> Run`（审批解锁回调）
   - 审批通过后，Decision 通过 unlock port 触发 Run 恢复。

3. `Decision/Run -> Evidence`
   - 两者只写 outbox。
   - 由 HTTP 层统一同步到 Evidence（避免内核互相直接耦合）。

### 3.2 核心协同接口

- Run 依赖的 `DecisionPort`（`internal/run/types.go`）：
  - `IsDecisionConfirmedWithOwner`
  - `GetDecisionReference`
  - `ValidateExecutionReceiptForRun`

- Decision 依赖的 `RunStatePort`（`internal/decision/types.go`）：
  - `GetRunState`

- Decision 审批解锁依赖的 `ApprovalRunUnlockPort`：
  - `DispatchApprovalRunUnlock`

## 4. 关键数据对象（先理解这些）

### 4.1 Decision 侧关键对象

1. `RuntimeDecisionRequest`
   - 运行时裁决输入，包含：risk/effect/freeze/dcu/feature 信号等。

2. `RuntimeDecisionResponse`
   - 输出决策（allow/require_approval/review_required/fail_closed 等）+ reason codes + obligations。

3. `PendingDecision`
   - 双确认中间态：先 pending，Run 推进成功后 confirmed。

4. `DispatchTicket`
   - 调度准入发放的执行票据（资源上限、绑定 run/step/tenant、过期时间）。

5. `StoredDecision`
   - 决策历史版本存储，支持 history/version 查询。

### 4.2 Run 侧关键对象

1. `RunView`
   - run 当前状态真值（state/state_version/current_step_seq_id/root 等）。

2. `StepIntegrity`
   - 每个 step 的完整性记录（prev hash + decision hash + in/out hash + step hash）。

3. `Continuation`
   - park/resume 令牌（type + seq + expire + status）。

### 4.3 Evidence 侧关键对象

1. `CanonicalEvent`
   - 统一证据事件模型（schema/source/run/decision/payload hash/tier/grade）。

2. `DecisionGraph`
   - 节点与边，支持根因路径分析。

3. `DecisionLog`
   - 决策摘要日志（含版本、来源事件、完整性分数）。

4. `ReplayPack` / `RootCausePack`
   - 回放包与排障包导出。

## 5. 端到端主流程（数据如何跑通）

下面按“最常见的一条业务链”讲。

### 5.1 链路 A：Runtime 决策

1. 客户端调用 `POST /v1/decision/evaluate-runtime`
2. Router 把请求给 `decisionSvc.EvaluateRuntime`
3. Decision 执行顺序（简化）：
   - 输入归一化
   - 生成 decision id / frozen hash
   - 校验 risk/effect/decision graph
   - 解析 feature signal contract（scope + key precedence）
   - 校验合同字段/信任约束（freshness/drift/producer/schema）
   - DCU 预算校验
   - freeze 白名单校验
   - policy phase 评估 + obligations 合并
   - soft failure guard
   - 持久化决策 +（如需）创建 pending decision + 发事件
4. Router 调 `syncEvidenceFromKernelOutboxes`，把 decision outbox 入证据库。
5. API 返回 runtime decision response。

### 5.2 链路 B：调度准入 + 票据

1. 调用 `POST /v1/decision/evaluate-schedule-admission`
2. Decision 根据 quota/priority/vip/preemptable 评估。
3. 允许时签发 `DispatchTicket`，绑定 `run_id + step_id + tenant_id + allowed_resources + expiry`。
4. 返回 decision hash + ticket。

### 5.3 链路 C：Run 推进（真正执行前门禁）

1. 调用 `POST /v1/runs/{run_id}/advance`
2. Run 执行顺序（关键）：
   - 校验 run 状态与版本
   - 校验 step seq 单调递增
   - 从 Decision 取 `DecisionReference`
   - 校验 confirmed（owner-aware）
   - 校验 decision hash stable + 绑定一致
   - 校验 decision value 可执行（只允许 allow/require_approval）
   - 调 Decision 校验 execution receipt（包含资源上限）
   - 执行 obligations（limit_param/attach_tag/require_template/emit_audit）
   - 通过后才推进状态 + 写 step integrity + 发 outbox
3. Router 同步 Run outbox 到 Evidence。

### 5.4 链路 D：审批闭环

1. `POST /v1/approval/cases` 创建审批单。
2. `POST /v1/approval/cases/{case_id}/decision` 审批通过/拒绝。
3. Decision 处理：
   - 更新 approval case 状态
   - 更新 pending decision 状态
   - 必要时 dispatch run unlock
   - unlock 失败进入 safeguard hold guard
   - 记录审批有效延迟指标
4. Decision 事件同步到 Evidence，形成可审计轨迹。

### 5.5 链路 E：Evidence 汇聚与取证

1. Decision/Run 各自产生 outbox 事件。
2. Router 的 `syncEvidenceFromKernelOutboxes()` 做映射后调用 `evidenceSvc.IngestEvent`。
3. Evidence 内核：
   - 去重（event_id）
   - canonicalize
   - 根据 tier/backpressure 决定 drop/degrade/accept
   - 更新 graph/log/ledger/integrity
4. 查询导出：
   - run evidence summary
   - decision graph
   - root cause pack
   - replay pack
   - archive export
   - integrity verify

## 6. 从零跑通：最小操作手册

### 6.1 启动

```bash
go run ./cmd/agent-infra
```

默认监听 `:8080`。

### 6.2 创建 run

```bash
curl -sS -X POST http://127.0.0.1:8080/v1/runs \
  -H 'Content-Type: application/json' \
  -d '{
    "request_id":"demo-run-1",
    "tenant_id":"tenant-a",
    "workflow_id":"wf-demo",
    "workflow_version":"v1",
    "risk_tier":"low",
    "input_payload_ref":"obj://input/1",
    "snapshot":{
      "policy_bundle_id":"pb-1",
      "model_profile_id":"mp-1",
      "dependency_bundle_id":"db-1",
      "skill_bundle_set_hash":"skill-hash-1",
      "context_policy_space_hash":"cps-1"
    }
  }'
```

记录返回的 `run_id`。

### 6.3 运行时裁决

```bash
curl -sS -X POST http://127.0.0.1:8080/v1/decision/evaluate-runtime \
  -H 'Content-Type: application/json' \
  -d '{
    "request_id":"req-demo-rt-1",
    "tenant_id":"tenant-a",
    "workflow_id":"wf-demo",
    "run_id":"<RUN_ID>",
    "step_id":"step-1",
    "risk_tier":"low",
    "effect_type":"read",
    "phase":"PRE_TOOL",
    "approval_system_available":true,
    "policy_engine_available":true,
    "feature_version":"fv-1",
    "feature_evidence_ref":"evidence://feature/fv-1",
    "feature_producer_id":"feature-producer-default",
    "dcu_input":{"feature_reads":1,"rule_evals":1,"dependency_calls":0,"conflict_resolutions":0},
    "freeze":{"frozen":{
      "context_candidates_snapshot_ref":"ctx-1",
      "policy_bundle_snapshot_ref":"pb-1",
      "feature_snapshot_id":"fs-1",
      "approval_routing_snapshot_ref":"ar-1",
      "quota_snapshot_ref":"qs-1",
      "scheduler_admission_input_snapshot_ref":"sa-1"
    },"dynamic_used":["trace_tags"]}
  }'
```

记录 `decision_id` 与 `decision_hash`。

### 6.4 确认 run 推进许可（双确认）

```bash
curl -sS -X POST http://127.0.0.1:8080/v1/decision/confirm-run-advance \
  -H 'Content-Type: application/json' \
  -d '{
    "decision_id":"<DECISION_ID>",
    "run_id":"<RUN_ID>",
    "step_id":"step-1",
    "success":true
  }'
```

### 6.5 申请调度准入并拿 ticket

```bash
curl -sS -X POST http://127.0.0.1:8080/v1/decision/evaluate-schedule-admission \
  -H 'Content-Type: application/json' \
  -d '{
    "request_id":"adm-1",
    "run_id":"<RUN_ID>",
    "step_id":"step-1",
    "tenant_id":"tenant-a",
    "risk_tier":"low",
    "requested_resources":{"tokens":10},
    "quota_remaining":{"tokens":100}
  }'
```

记录返回 `ticket.ticket_id`。

### 6.6 推进 run（执行面硬门禁）

```bash
curl -sS -X POST http://127.0.0.1:8080/v1/runs/<RUN_ID>/advance \
  -H 'Content-Type: application/json' \
  -d '{
    "expected_state_version":1,
    "step_id":"step-1",
    "step_version":"1",
    "idempotency_key":"idem-1",
    "decision_ref":{
      "decision_id":"<DECISION_ID>",
      "decision_hash":"<DECISION_HASH>",
      "decision_hash_kind":"stable"
    },
    "execution_receipt_ref":"ticket://<TICKET_ID>",
    "execution_used_resources":{"tokens":8},
    "incoming_step_seq_id":1,
    "input_hash":"in-hash-1",
    "output_hash":"out-hash-1",
    "next_state":"RUNNING"
  }'
```

如果票据绑定或资源上限不对，会返回 `RUN_EXECUTION_RECEIPT_REJECTED`。

### 6.7 查看证据是否跑通

```bash
# 手动触发一次同步（可选，很多 API 已自动 sync）
curl -sS -X POST http://127.0.0.1:8080/v1/evidence/sync-kernel-outboxes

# 看 run 证据汇总
curl -sS http://127.0.0.1:8080/v1/evidence/runs/<RUN_ID>

# 看决策图
curl -sS http://127.0.0.1:8080/v1/runs/<RUN_ID>/decision-graph

# 看 replay pack
curl -sS "http://127.0.0.1:8080/v1/runs/<RUN_ID>/replay-pack?mode=minimal"
```

## 8. 常见误区（看代码时最容易踩）

1. `CanExecuteSideEffect` 不是最终执行门。
   - 最终必须看 `CanExecuteSideEffectFinal`（Decision + Run 状态 + receipt/ticket）。

2. `review_required` 不是可执行决策。
   - Run 执行面只接受 `allow` / `require_approval`。

3. `require_template` 不是字符串包含。
   - 现在是结构化模板证据提取与精确匹配。

4. Decision/Run 不直接写 Evidence。
   - 通过 outbox + Router sync 进入 Evidence。

5. 当前内核默认是内存状态。
   - 重启后状态会丢；持久化一致性属于下一阶段演进。
