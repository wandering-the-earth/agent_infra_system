# Agent Infra 团队开发守则（必须遵守）

更新时间：2026-04-03
适用范围：`/home/wandering/learn/agent/doc` 全部设计与后续实现

## 1. 目标

1. 降低线上事故率，保证变更可控、可回滚、可追责。
2. 保证“设计一致性”优先于“局部优化”。
3. 保证新成员可在最短时间按统一方式开发与发布。

## 2. 基本原则

1. 单一真值：主规范与子规范只能有一个最终裁决源。
2. 先证据后改动：没有评测与影响面报告，不得放行高风险变更。
3. 小步快跑：单次 PR 聚焦单一问题，不做无边界重构。
4. 失败可恢复：所有关键改动必须有回滚方案与 kill switch。
5. 可控性不等于正确性：任何“门禁通过”都不代表“绝对正确”，必须保留反证与复盘机制。
6. Eval 是证据不是终极真值：高风险发布必须做证据融合而非单指标放行。

## 3. 角色责任

1. `Owner`：对模块质量、SLO、上线结果负责。
2. `Reviewer`：只按规范与证据评审，不按个人偏好评审。
3. `Release Manager`：负责门禁执行与回滚决策。
4. `Oncall`：负责告警处置、事故分级、复盘闭环。

## 4. 分支与提交规范

1. 分支命名：`feature/<domain>-<short-desc>`、`fix/<domain>-<short-desc>`。
2. 提交信息：`<type>(<scope>): <summary>`，`type` 限定为 `feat|fix|refactor|docs|test|chore`。
3. 禁止把不相关变更混入同一 PR。
4. 禁止无说明的“纯格式化大改”进入核心模块。

## 5. 代码与接口规范

1. Core 域主语言为 `Go`，不得在 Core 同域引入第二主语言实现重复能力。
2. 跨服务调用必须使用版本化契约（OpenAPI/Proto），禁止隐式字段依赖。
3. 任何外部依赖接入必须经过适配层，不得直接进入核心裁决链。
4. 高风险写操作必须声明幂等键、补偿策略与审批策略绑定。
5. 调度执行可外包，但调度治理裁决（admit/reject/VIP/风险阻断/恢复资格）不得外包。

## 6. 评审与门禁规则

1. PR 合并前必须通过：静态校验、单测、关键回归、策略门禁、评测门禁。
2. 涉及策略变更必须附：`policy replay report` 与误伤评估。
3. 涉及上下文编译器变更必须附：deterministic 回归结果与降级路径验证。
4. 涉及调度/配额变更必须附：容量估算与 blast radius 报告。
5. 涉及 Decider 链路变更必须附：`Decision Causality Graph` 完整性报告（关键节点/关键边覆盖率）。
6. 涉及 skill 注入变更必须附：签名校验结果与越权覆盖检查报告。

## 7. 测试与评测要求

1. 单元测试覆盖核心分支与失败路径。
2. 集成测试覆盖：tool park/resume、审批恢复、超时重试、补偿流程。
3. 发布前必须运行最小评测套件（quality/safety/cost/latency）。
4. 新增能力默认先灰度，灰度期间必须有可观测看板与回滚开关。

## 8. 发布与回滚

1. 发布必须携带 `execution_snapshot_version` 与依赖冻结信息。
2. 运行中实例恢复必须遵循“原版本优先”兼容策略。
3. 任何变更均需定义回滚触发条件、执行命令与预期恢复时长。
4. 未完成回滚演练的高风险功能不得全量。

## 9. 事故与复盘

1. 事故分级：`P0/P1/P2/P3`，按等级定义响应 SLA。
2. P0/P1 必须在 24 小时内提交初步复盘，72 小时内提交闭环行动。
3. 复盘必须包含：根因、检测缺口、决策失误、工程补丁、流程改进。
4. 同类事故重复出现两次，必须升级为架构级整改项。

## 10. 文档与一致性治理

1. 设计变更必须同步更新主文档与相关子规范。
2. 引用路径必须使用 `agent/doc/*`，禁止保留历史路径。
3. 任一术语（如 Context Compiler、Policy Bundle、Eval Gate）只允许一个定义源。
4. 每次发布前必须执行“文档一致性检查”（路径、术语、规则冲突）。
5. 新增决策链路不得以隐式字段耦合替代因果节点，必须显式记录到 `decision_graph`。

## 11. 新功能接入流程

1. 提交 RFC（问题、方案、优缺点、证据、风险、回滚）。
2. 标记层级（Core/Extended/Experimental）与阶段（Phase1/2/3）。
3. 完成最小实现与不做清单。
4. 通过门禁后先灰度，再按证据决定是否全量。

## 12. 禁止项

1. 禁止绕过 Policy/Approval 直接执行高风险动作。
2. 禁止在生产环境直接试验 Experimental 能力。
3. 禁止无审计证据的热修复进入主链路。
4. 禁止跳过评测门禁放行结构性变更。
